Dragging a pane divider in TUIOS trailed the mouse. Not by a frame. By enough
that letting go of the button meant waiting to see where the layout landed, and
it got worse the longer the drag went on.

That last detail is the entire diagnosis, and I want to start there because I
nearly skipped past it.

## A slow frame and a backlog do not look the same

If frames are simply slow, the lag is constant. Every frame costs the same, you
are always the same amount behind, and dragging for five seconds is no worse
than dragging for one.

If events arrive faster than they can be serviced, the lag grows for as long as
the input keeps coming. That is a backlog, and it is a different bug with a
different fix.

TUIOS was in the second case. Every mouse motion event cleared the frame-skip
flag, so each one composed a full frame. A frame during a tiling resize costs
3.3 to 6.7 ms, which caps the drain rate at roughly 150 to 300 events per
second. A drag emits one event for every cell the pointer crosses. Anything
faster than that accumulated.

<BacklogChart />

The fix is not to make the frame faster. It is to stop composing one per event.
Motion events during an active drag now redraw at most once per normal frame
interval.

The constraint that makes this safe is that nothing is dropped except redundant
intermediate frames. The handler still applies every event's geometry before it
makes the draw decision, so the model always holds the newest pointer position.
The interaction tick flushes a skipped position within one tick, and release is
not a motion event, so the final position always draws. The layout settles
exactly where the mouse was let go.

## The tooling that made this cheap

None of this required anything exotic. Go ships the whole loop, and the reason I
tested four hypotheses instead of committing to the first one is that testing
each was a couple of minutes.

The benchmark that produced the table above lives in the repository next to the
code it measures, and it drives the real message handler and the real view:

```go
func BenchmarkResizeMotionFrame(b *testing.B) {
	app.SetInputHandler(HandleInput)

	for _, shared := range []bool{false, true} {
		for _, n := range []int{2, 4, 9} {
			b.Run(fmt.Sprintf("windows-%d/shared-%v", n, shared), func(b *testing.B) {
				m := benchResizeOS(b, n)
				startX, startY := m.ResizeStartX, m.ResizeStartY

				b.ReportAllocs()
				b.ResetTimer()
				i := 0
				for b.Loop() {
					// Sweep the pointer by a cell so every event is a real
					// geometry change, not a no-op.
					dx := i%6 - 3
					_, _ = m.Update(motionAt(startX+dx, startY))
					_ = m.View()
					i++
				}
			})
		}
	}
}
```

Two things about that benchmark are deliberate. It goes through `Update` and
then `View`, because `View` is what Bubble Tea runs after every `Update` and the
per-event cost the user pays includes both. And the pointer sweeps rather than
sitting still, because a motion event that does not change geometry takes an
early return and would benchmark a path nobody is on.

The commands:

```
# One case, with allocations, repeated so variance is visible.
go test -run XXX -bench BenchmarkResizeMotionFrame -benchmem -count=5 ./internal/input/

# Expensive benchmarks are pinned to a fixed iteration count rather than a
# duration, so runs are comparable across machines and across days.
go test -run XXX -bench BenchmarkCompositorGetCanvas -benchtime=200x ./internal/app/
```

`-run XXX` is the small habit that matters most in that line. Without it the
package's whole test suite runs before the benchmark does, which is slow and,
for anything that touches shared state, occasionally not harmless.

For the things a benchmark cannot see, TUIOS serves pprof from the running
process. `cmd/tuios/run.go` imports `net/http/pprof` for its side effect and
puts the server behind a flag:

```go
import _ "net/http/pprof" // registers /debug/pprof handlers on http.DefaultServeMux

if pprofAddr != "" {
	runtime.SetBlockProfileRate(10000) // one sample per ~10us blocked
	runtime.SetMutexProfileFraction(100)
	go func() {
		srv := &http.Server{Addr: pprofAddr, ReadHeaderTimeout: 5 * time.Second}
		if err := srv.ListenAndServe(); err != nil && !errors.Is(err, http.ErrServerClosed) {
			log.Printf("pprof server error: %v", err)
		}
	}()
}
```

```
tuios --pprof localhost:6060
go tool pprof -http=: http://localhost:6060/debug/pprof/profile?seconds=20
```

Block and mutex profiling are sampled rather than exhaustive here on purpose. A
rate of 1 samples every event and adds enough overhead under load that the
contention data stops describing the program you were trying to measure. Output
goes to a log rather than stdout, because a profiler that prints to stdout will
happily corrupt the TUI it is profiling.

The soak test in `e2e/tui/soak_test.go` reads the same endpoints from outside
the process, polling `/debug/pprof/goroutine?debug=1` and
`/debug/pprof/heap?debug=1` while it drives the binary through mixed activity.
Goroutine count is the sharper of the two signals: heap in use drifts for
reasons that are not leaks, and a goroutine count that only goes up is much
harder to argue with.

## Four hypotheses, two wrong

I had four candidate explanations for the resize lag before measuring anything.
Two survived.

**Whole-tree ratio re-derivation is expensive.** Plausible: it walks every
window in the BSP tree and re-derives every split ratio, on every motion event.
Measured at 6 to 86 us per event against a multi-millisecond frame. About one
percent. Optimising it would have bought nothing, and I would have shipped a
change that felt like progress and moved no numbers.

**Damage tracking is not buying anything.** This one is the honest failure in
the set, because the mistake was not in the measurement, it was in my reading of
it. I looked at a benchmark comparing one dirty window against all windows
dirty, saw effectively identical timings, and concluded that damage tracking was
not earning its complexity. The benchmark was running at one window. At one
window, "one dirty" and "all dirty" are the same case. Identical timings were
not evidence of anything at all.

At window counts where the two differ, they differ:

I did not catch that on my own. I passed it on as a finding first. That is worth
saying out loud, because the failure mode was not sloppy measurement, it was
trusting a number without checking what the two sides of the comparison actually
were.

**The tick throttle is limiting the flood.** Real, but backwards. `SlowTickCmd`
governed the periodic tick and dropped it to 30 FPS during interaction, while
motion events drove their own renders regardless of the tick. It never limited
anything. It cost smoothness and bought nothing, and it was deleted along with
`InteractionFPS` once nothing referenced it.

**PTY resize is happening per motion event.** This one was already correct in
the code, deferred to drag completion. Worth confirming rather than assuming: a
`TIOCSWINSZ` on every motion event would have been catastrophic and would have
produced exactly the same symptom, so ruling it out early was cheap insurance
against a very expensive wrong turn.

## Fixing the top cost promotes the next one

After coalescing, the render ran once per frame interval but the handler still
ran on every event, so the handler became the new floor. And a gap that had been
invisible underneath a multi-millisecond frame was suddenly the dominant term.

Allocation counts said the same thing more bluntly. At nine windows: 10,665 B in
8 allocations with shared borders off, 57,238 B in 34 allocations with them on,
and 9,846 B in 8 after the fix. `-benchmem` is often more diagnostic than the
timing column, because allocation counts are stable across machines and load in
a way nanoseconds are not.

The cause was `SyncBSPTreeFromGeometry` running on every motion event,
rebuilding a geometry map over every window and re-deriving every ratio in the
whole tree. That is the same function I had correctly measured as negligible an
hour earlier. It was one percent of a multi-millisecond frame. Once the frame
was gone, it was most of what was left.

A performance conclusion has a shelf life. "Negligible" is a statement about a
ratio, and you just changed the denominator.

The fix mirrors the render coalescing: the sync only exists so the separator
overlay tracks the pointer, so it only has to be current on frames that are
actually composed. Motion marks the tree dirty, and the frame decision flushes
it. The sync on mouse release stays unconditional, because a retile rebuilds
from the tree's ratios rather than from the window rectangles, and a stale tree
means the next retile silently discards the resize the user just performed.

## The performance fix caused a rendering bug

Deferring the sync produced a visible artefact: dragging a divider with shared
borders flashed a red separator beside the highlighted one, reading as an
afterimage of where the divider had just been.

`renderSeparatorOverlay` mixes two sources. It collects divider positions from
the BSP tree, and it takes the highlight from the focused window's perimeter,
which is live window geometry. Deferring the sync made those two disagree for as
long as a sync was outstanding. A frame drawn in that window put the divider
where the drag had already left, and since that column was no longer on the
focused pane's perimeter, it lost the focus colour and came out red.

The interesting part is where I put the flushes, which was wrong. I placed them
on the motion handler's render decision and on the interaction tick, because
those are the paths that change geometry. Neither covers a frame composed for
some other reason, and during a real drag there is always another reason: PTY
output. Terminals produce output continuously, and `PTYDataMsg` composes frames.
That is why the artefact appeared constantly in use and needed interleaved PTY
output to reproduce in a test at all.

The flush belongs in `View()`, immediately before `composeFrame()`. That is
where the constraint actually lives. Geometry may be applied whenever, but the
ratios have to match it on any frame that reaches the screen. Frames served from
the skip cache still cost nothing, so the per-event work the deferral bought is
unchanged, flat at 7.4 us and 9.8 KB from four windows to nine.

The test drives a shared-borders drag interleaved with PTY output and scans
every composed frame for the focused pane's edge column, failing if it carries
the unfocused colour or no separator at all. Before the fix it reported 18 of 24
mid-drag frames wrong, each with a red divider one to four columns from the pane
edge. After, 0 of 24. Removing the flush again with the test in place fails 24 of
24, which is the only version of "the test works" I trust.

## Where micro-optimisation cannot help

Everything above is tuning. Sometimes the shape of the code is the problem, and
no amount of tuning reaches it.

### A quadratic hiding behind a passing benchmark

The web client for [sip](https://github.com/Gaurav-Gosain/sip) had typing
latency bad enough to be unusable, and every benchmark it had looked fine.

`getLine(row)` walked the entire viewport and sliced one row out of the result.
The render loop called `getLine` once per damaged row. A frame was therefore
O(rows squared times cols) crossings into wasm, at roughly eight crossings per
cell.

At the 80x24 and 120x40 grids the benchmarks used, that is invisible. At the
193x56 a maximised terminal actually runs, it is crippling. Both the canvas2d
and the WebGL renderers were reported as unusably slow while every benchmark
stayed green, which is a good argument for benchmarking at the sizes people use
rather than the sizes that are convenient.

The fix is not a faster inner loop. `getViewport` memoizes its cell pool,
invalidated only where the grid can actually change, so a frame performs exactly
one walk however many rows it reads. Measured in Chromium at 193x56, per
rendered frame while typing: canvas2d 30.0 ms to 6.0 ms in `getViewport`, WebGL
187.0 ms to 11.6 ms.

No constant-factor tuning of that inner loop closes a quadratic gap. You could
have made every crossing twice as fast and still had an unusable terminal at
full screen.

That fix then promoted the next cost, exactly as in TUIOS. With the walk down to
one per frame, the cost of the walk itself became the term that mattered: it was
constructing about six DataViews and one Uint8Array per cell, roughly 80,000
typed-array constructions per frame at 207x55. Caching the views by buffer
identity took the viewport walk from 6.3 ms to 1.3 ms in Chromium and 11.64 ms
to 3.32 ms in Firefox, and typed arrays per walk from 79,918 to 1.

And then the next one after that: `markClean()` existed on the wasm terminal and
was called from nowhere, so every row reported dirty forever and a keystroke
that changed one cell was serviced by re-reading all 11,385 of them. Clearing
damage after each frame took the typing-path read from 1.4 ms to effectively
zero in Chromium and 3.90 ms to 0.12 ms in Firefox. The full repaint number did
not move, 11.5 ms to 11.2 ms, and should not have: on a full repaint every row
is genuinely dirty.

Three fixes, each of which only became the bottleneck once the previous one was
gone.

### A freeze no profiler would find

TUIOS would occasionally lock solid at zero CPU with a live shell prompt in a
pane. Nothing to profile: there is no hot loop, because nothing is running.

`renderTerminal` held a window's I/O read lock for its cell walk, then called
`getRealCursor`, which takes the focused window's read lock again. When the
rendered window is the focused one, that is the same `sync.RWMutex` acquired
twice on one goroutine.

Go's `RWMutex` is not reentrant for readers, and deliberately so. Once a writer
calls `Lock`, subsequent `RLock` calls block, so that writers cannot be starved
by a stream of readers. The PTY reader takes the exclusive side on every chunk
of shell output, so it regularly queued between the two acquisitions. The UI
goroutine then parked on its second `RLock` behind a writer that was itself
waiting for the first `RLock` to be released. Neither could proceed.

That is a goroutine dump, not a CPU profile. `SIGQUIT` the process, or read
`/debug/pprof/goroutine?debug=2`, and the answer is right there in the stacks.
The fix is to query the cursor before the lock is taken, since the value only
decides whether the cell loop paints a fake cursor and does not need the lock the
walk holds.

The same lock-ordering inversion existed between the graphics passthrough locks
and the window I/O lock, and got the same treatment: snapshot the values before
the refresh so the pair is only ever acquired in one order. The non-reentrancy
rule is now recorded at the lock definition, so the next caller reaches for the
lock-free variant or hoists the read rather than rediscovering this.

## An optimisation I measured and threw away

Grapheme cluster width is a real correctness problem in a terminal, and
ghostty-vt gets it right. So I evaluated it as a runtime backend, built an
offline width table from it, applied the table, and measured.

It changed nothing.

The reason is that cluster advance is UAX 29 join accumulation rather than
per-codepoint width lookup. A width table, however correct its entries, answers
a question that is not the one being asked. The work went in the bin, xterm.js
plus the official graphemes addon stayed, and ghostty-vt now serves as an oracle
for validating width tables offline rather than running in production. A
45-case grapheme corpus is measured against real ghostty-vt wasm with mode 2027
clustering: 39 agree, 6 diverge in documented degenerate cases, and those six are
asserted as expected values rather than filed as bugs.

I count that as a good outcome. It cost a day and it retired a hypothesis I would
otherwise still be carrying around.

## What a benchmark cannot tell you

Two limits I state in the repositories rather than quietly hoping nobody asks.

[vtgl](https://github.com/Gaurav-Gosain/vtgl) benchmarks run under headless
Chromium, where WebGL runs on SwiftShader and rasterises on the CPU. Only the
CPU column, the draw-call counts and the allocation figures transfer to real
hardware. No frame rate is claimed anywhere, because the number that harness
produces for frame rate is a number about SwiftShader.

TUIOS still has a fixed frame floor of roughly 0.7 to 1.3 ms, from full-screen
canvas composition that runs regardless of how many windows are dirty. Damage
tracking reduces per-window work and does not touch this floor.
`BenchmarkCompositorGetCanvas` puts a call at 1.2 to 2.0 ms depending on window
count and dirtiness, and `BenchmarkRenderScalingWithCells` shows the scaling
plainly: 74,728 ns at 80x24, 207,331 ns at 120x40, 507,587 ns at 207x55 and
907,111 ns at 280x70. It no longer matters for resize, being well inside a 16.6
ms budget, but it caps the overall frame rate and it will be the next thing to
go when frame rate is what is wanted.

A known, quantified, deliberately unfixed limit is a much better position than an
unknown one.

## The short version

Measure the shape, not just the size. A cost that grows with time is a backlog. A
cost that grows with window count is whole-tree work. A cost that grows with the
square of the row count is an algorithm you have to replace rather than tune. The
absolute numbers told me much less than how they moved.

Check what your benchmark actually compares before you believe its answer. One
dirty window against all dirty windows is not a comparison when there is one
window.

Every conclusion is dated. Fixing the dominant cost promotes the next one, and
the thing you correctly measured as one percent of a frame can be most of the
frame an hour later.

And verify fixes by breaking them. Reverting a fix and confirming the test fails
caught five cases in this run where something was believed fixed and was not.

The [architecture documentation](/docs/architecture) covers how the compositor
and the BSP tree fit together, if you want the structure that all of this is
sitting on top of.
